# Change Reports

Change reports are generated at the end of every BIOME build session. They provide a complete audit trail of what was built, changed, and deployed — across applications, databases, and infrastructure.

## Purpose

- **ARB (Architecture Review Board)** — review what changed and why before approving
- **Team awareness** — anyone can see what happened in a session without reading code diffs
- **Audit trail** — every database migration, Edge Function, and architecture decision is documented
- **Onboarding** — new team members can read through reports chronologically to understand how the platform evolved

## File format

Each report is a JSON file named by date:

```
change-report-YYYY-MM-DD.json
```

If multiple sessions happen on the same day, append a suffix:

```
change-report-YYYY-MM-DD-02.json
```

## How to read a report

Each report contains these top-level sections:

| Section | What it tells you |
|---|---|
| `summary` | One-sentence overview of the session |
| `applications.created` | New apps scaffolded, with stack, files, and dependencies |
| `applications.modified` | Changes to existing apps |
| `database_changes` | Migrations applied to Supabase projects, with per-migration details |
| `health_check_results` | Snapshot of infrastructure health at end of session |
| `pending_actions` | Things that need human action (secrets, dashboard toggles, etc.) |
| `architecture_decisions` | Key decisions made and why |

## Generating a report

Reports are generated by Wren (the build assistant) at the end of each session. If a session ends without a report, request one:

```
Generate Change Report please.
```

The report will be saved to `docs/change-reports/` and should be committed with the session's changes.
